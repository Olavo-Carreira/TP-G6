# P2P Privacy-Preserving Auction System

Group project for a Distributed Systems / Applied Cryptography course at Universidade do Minho. The goal was to design and implement a peer-to-peer auction system in which announcements and bids are distributed to all participants while preserving anonymity, authenticity, integrity, non-repudiation and trusted timestamping, with mutual identity disclosure limited to the seller and the winning bidder at the end of each auction.

## Project Structure

### `DS - Group project.pdf`
The original assignment specification, describing the system requirements (auction announcements, bidding process, winner revelation), the mandatory security properties (anonymity, authenticity, integrity, non-repudiation, trusted timestamping, selective identity disclosure), the architecture constraints, and the deliverables.

### `src/server/server.py`
A Flask-based HTTPS server that plays the role of the trusted/central component allowed by the assignment. It exposes endpoints for user registration and public-key lookup, peer discovery/announcement (`/peers/announce`, `/peers/list`), and a trusted timestamping service (`/timestamp`, `/timestamp/verify`) that signs a hash together with a timestamp so bids can be ordered and ties resolved by earliest submission. It also exposes basic health and stats endpoints. TLS is enabled using the certificate/key pair generated by `generate_cert.py`.

### `src/generate_cert.py`
Generates a self-signed RSA certificate and private key (`server_cert.pem` / `server_key.pem`) used to serve the server over TLS, protecting the confidentiality and integrity of client-server communication as required by the assignment.

### `src/server_cert.pem` / `src/server_key.pem`
The self-signed TLS certificate and corresponding private key used by the server (generated via `generate_cert.py`).

### `src/config.py`
Shared configuration module for the system (currently empty/placeholder).

### `src/client/crypto_utils.py`
Core cryptographic primitives used throughout the client: RSA keypair generation, key (de)serialization, digital signature creation/verification, and symmetric encryption helpers used to encrypt identity-reveal messages so only the intended recipient (seller or winner) can read them.

### `src/client/ring.py`
Implementation of ring signatures. Allows a user to sign an announcement or bid on behalf of a group of possible signers (the "ring") without revealing which member actually produced the signature, providing anonymity for sellers and bidders while still proving the signer is a legitimate registered participant.

### `src/client/commitement.py`
Implements a commitment scheme (hash of the bid/reserve value plus a random nonce). Sellers and bidders first publish a commitment, later revealed and verified, so that bid/reserve values cannot be tampered with or altered after submission while remaining hidden until reveal.

### `src/client/announcement.py`
Defines the `AuctionAnnouncement` structure (item description, reserve-price commitment, start/end time), signed with a ring signature and timestamped by the trusted timestamping service, so the seller's identity stays hidden among the ring of possible signers.

### `src/client/bid.py`
Defines the `Bid` structure, combining a value commitment, a ring signature over the bid, and a trusted timestamp, so bids are anonymous, authenticated as coming from a legitimate participant, tamper-evident, and orderable in time.

### `src/client/reveal.py`
Defines the `IdentityReveal` structure used at the end of an auction to let the seller and the winning bidder exchange their identities (encrypted so only the counterpart can decrypt), while every other participant remains anonymous.

### `src/client/winner.py`
Defines the `AuctionResult` structure, recording the winning bid, winning amount, reserve price and number of valid bids once an auction concludes.

### `src/client/status.py`
Defines the `AuctionStatus` enum used to track the lifecycle of an auction (`announced`, `active`, `bidding_closed`, `revealing`, `completed`).

### `src/client/block.py`
Defines the `Block` structure used by the local blockchain (index, previous hash, timestamp, transactions, nonce, SHA-256 hash), providing an append-only, tamper-evident log.

### `src/client/blockchain.py`
Implements the `Blockchain` class that stores announcements, bids and reveals as transactions in a chain of blocks, keeping track of pending transactions, seen transaction hashes (duplicate protection) and used ring-signature key images, giving the system integrity and a shared source of truth across peers.

### `src/client/network.py`
Handles peer-to-peer networking: discovering and connecting to other nodes, and broadcasting/receiving announcements, bids, reveals and blockchain data among peers.

### `src/client/node.py`
Represents a running peer (`AuctionNode`): holds the user's identity/keys, ring keys, blockchain, active peers and auctions, and coordinates the P2P and cryptographic modules above.

### `src/client/manager.py`
Higher-level orchestration logic that ties together announcements, bidding, reveal and winner-determination flows on top of a node.

### `src/client/cli.py`
Command-line interface utilities (menus, tables, colored/boxed output, password prompts, node status display) used to interact with the system as an end user.

### `src/client/my_secrets.json`
Local, user-specific persisted data (e.g., commitments' nonces/values needed to reveal a bid or reserve price later). Contains sensitive local state and should not be shared or committed with real data.

## Frameworks and References

- [Python `cryptography`](https://cryptography.io/) — RSA key generation, digital signatures, X.509 certificates, symmetric encryption.
- [Flask](https://flask.palletsprojects.com/) — HTTPS server exposing the trusted registry/timestamping endpoints.
- Python standard library — `hashlib` (SHA-256 hashing for commitments and blockchain), `ssl`, `json`, `time`.
- Cryptographic concepts applied: ring signatures (signer-ambiguous authentication), commitment schemes (hide-then-reveal integrity), trusted timestamping (signed hash + time), and a simple blockchain (hash-linked ledger of transactions).

## Disclaimer

This project was developed for academic purposes as part of a course assignment at Universidade do Minho. It is a proof-of-concept implementation intended to demonstrate the application of cryptographic techniques to a peer-to-peer auction scenario and has not been audited or hardened for production use.
